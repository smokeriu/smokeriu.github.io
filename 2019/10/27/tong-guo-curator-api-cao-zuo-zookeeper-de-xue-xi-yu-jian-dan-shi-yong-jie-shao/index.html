通过Curator API操作zookeeper的学习与简单使用介绍

Zookeeper
Zookeeper
发布日期:   2019-10-27

主要记录些使用Curator的学习记录，方便随时查阅。

Curator

Curator框架在zookeeper原生API接口上进行了包装，解决了很多ZooKeeper客户端非常底层的细节开发。提供ZooKeeper各种应用场景(recipe， 比如：分布式锁服务、集群领导选举、共享计数器、缓存机制、分布式队列等)的抽象封装，实现了Fluent风格的API接口,是最好用，最流行的zookeeper的客户端。包括以下特性：

  1. 自动化的连接管理。
  2. 简化了原生的ZooKeeper的方法，事件等
  3. Curator初始化之后会一直对zk连接进行监听，一旦发现连接状态发生变化将会作出相应的处理。
  4. 内部实现了诸如Session超时重连，Watcher反复注册等功能，

更多相关信息请前往：http://curator.apache.org

使用概述：

curator是Fluent风格API，创建会话方式与原生的API创建方式区别很大。
Curator创建客户端为CuratorFramework，是由CuratorFrameworkFactory工厂类来实现的，注：CuratorFramework是线程安全的，要连接的每个ZooKeeper集群只需要一个 CuratorFramework对象就可以了。

添加依赖：

<dependency>
  <groupId>org.apache.curator</groupId>
  <artifactId>curator-recipes</artifactId>
  <version>${curator.version}</version>
</dependency>

一般添加这个就够用了，这个依赖背后依赖了client和framework。
client是zookeeper client的封装，提供有用的客户端特性。
framework是api的高层封装，添加了连接管理，重试机制等。
本次使用版本：
curator：4.0.1
zookeeper：3.4.6（3.4版本需要排除zk的依赖包。）

获取client

总体而言，通过CuratorFrameworkFactory来获取Client对象。一种则是使用.builder()。另一种使用构造方法直接new，其实内部走的一条线：

/**
 * 参数：
 * @param connectString       list of servers to connect to（zk地址）
 * @param retryPolicy         retry policy to use（重试策略）
 * 可选参数：
 * @param sessionTimeoutMs    session timeout（会话超时时间）默认60000
 * @param connectionTimeoutMs connection timeout（连接超时时间）默认15000
 * @param namespace           每个curatorFramework可以设置独立命名空间，之后操作基于该命名空间。
 *                            比如操作/test/abc =>实际是操作：/zk1/test/abc.注意使用时前面不要加/号
 */
cf2 = CuratorFrameworkFactory.builder().connectString("hadoop:2181")
        .sessionTimeoutMs(60000)
        .connectionTimeoutMs(15000)
        .retryPolicy(retryPolicy)
        .namespace("test")
        .build();

//也可以采用常规的newClient的形式，不过无法支持命名空间。
//其实内部还是使用的builder，给出其中一个构造方法。
public static CuratorFramework newClient(String connectString, int sessionTimeoutMs, int connectionTimeoutMs, RetryPolicy retryPolicy)
{
    return builder().
        connectString(connectString).
        sessionTimeoutMs(sessionTimeoutMs).
        connectionTimeoutMs(connectionTimeoutMs).
        retryPolicy(retryPolicy).
        build();
}

上面代码中有使用到一个实例，retryPolicy，即重试策略。
Curator提供了多种重试策略，后边会进行一个简单介绍，选择合适的即可。
不过目前为止，我们已经拿到了client。这时候需要我们调用start()来使用他。程序结束时我们也应该要调用close()来关闭他。

cf2.start()
cf2.close()

重试策略简介

会给出构造方法即部分介绍，一般看看名字和参数就知道有啥区别了。
他们都是继承自SleepingRetry
常用的有：
ExponentialBackoffRetry
ExponentialBackoffRetry被BoundedExponetialBackoffRetry继承，参数作用都差不多，不做赘述。

/**
 * @param baseSleepTimeMs initial amount of time to wait between retries（每次重试会增加重试时间baseSleepTimeMs）
 * @param maxRetries max number of times to retry（最大重试次数）
 * @param maxSleepMs max time in ms to sleep on each retry（最大重试时间）
 *                   如果超过，将使用warn级别log，并使用最大重试时间
 */
public ExponentialBackoffRetry(int baseSleepTimeMs, int maxRetries, int maxSleepMs)

RetryForever（一直重试）

/**
 * @param retryIntervalMs    重试间隔时间
 */
public RetryForever(int retryIntervalMs)

RetryNTimes（重试N次）

/**
 * @param retryIntervalMs    重试间隔时间
 * @param n                    重试N次
 */
public RetryNTimes(int n, int sleepMsBetweenRetries)

剩下还有RetryOneTime（重试一次）、RetryUntilElaspsed（一直重试，知道超过指定时间）

常用的API

这边大家直接看代码注释。因为使用的是zookeeper3.4。部分3.5的特性不支持。

创建节点

可以允许递归创建，即父节点不存在时可以帮你创建父节点。

/**
 * 创建节点。
 * 常见节点类型：
 *      永久：PERSISTENT
 *      临时（当前连接有效）：EPHEMERAL
 */
@Test
public void create() throws Exception {
    cf2.create().creatingParentsIfNeeded()//允许递归创建
            .withMode(CreateMode.PERSISTENT)//持久绩点
            .withACL(ZooDefs.Ids.OPEN_ACL_UNSAFE)
            .forPath("/test2","data".getBytes());

}

获取节点数据：

/**
 * 获取节点信息
 * data会作为byte数组返回。
 * 节点信息则会放入stat中。
 */
@Test
public void getInfo() throws Exception{
    Stat stat = new Stat();
    byte[] bytes = cf2.getData()
            .storingStatIn(stat)//将节点信息传入stat中（和zk自带的stat一样）
            .forPath("/a");
    System.out.println(new String(bytes));
    System.out.println(stat.getNumChildren());
}

/**
 * 获取节点信息，并使用watcher
 * 默认只能监听本节点的修改、删除
 * 获取children可以用getChildren。
 * 初次运行节点不存在会报错。watcher只能监听一次。
 */
@Test
public void getInfoWithWatcher() throws Exception{
    Stat stat = new Stat();
    byte[] bytes = cf2.getData().storingStatIn(stat)
            .usingWatcher(new Watcher() {
                @Override
                public void process(WatchedEvent watchedEvent) {
                    switch (watchedEvent.getType()){
                        case NodeDataChanged:
                            System.out.println("change!");break;
                        case NodeDeleted:
                            System.out.println("node delete!");break;
                        case NodeChildrenChanged:
                            System.out.println("child!");break;
                        case NodeCreated:
                            System.out.println("Create!");break;
                    }
                }
            }).forPath("/test");
    System.out.println(stat.getVersion());
    System.out.println(new String(bytes));
    Thread.sleep(60*1000);
}

设置/修改节点数据

/**
 * 设置节点
 * 不指定会使用defaultData。默认为ip
 * 返回设置后的stat
 */
@Test
public void setNode() throws Exception {
    Stat stat = cf2.setData().withVersion(-1).forPath("/test");
    System.out.println(stat.getVersion());
}

删除节点

/**
 * 删除节点
 * deletingChildrenIfNeeded：子节点存在则删除子节点，再删除父节点。
 * guaranteed：如果删除失败，后端会一直尝试删除知道成功。
 */
@Test
public void deleteNode()throws Exception{
    Void aVoid = cf2.delete()
            .guaranteed()//可省略，保证删除节点（删除失败后段会一直删除知道成功）
            .deletingChildrenIfNeeded()//可省略，开启递归删除。
            .forPath("/test");
    System.out.println(aVoid);
}

Cache（watcher N次）

原生的zk，有个比较蛋疼的地方就是watcher只能使用一次，如果需要重复使用，常见的可以采用生产消费者模式来重复触发注册，不过还是比较繁琐的。
幸运的是，Curator自带了一套名为Cache的东西，能够帮助我们完成类似功能：
注：上面有使用过usingWatcher的方法，不过也是一次性的。

Cache一般有三种：NodeCache、PathChildrenCache、TreeCache。
NodeCache是监视当前节点。
PathChildrenCache则是监视该节点的子节点。（不能递归监视孙节点哦）
TreeCache则是前两者的结合，即监视当前节点+当前节点的子节点。
他们使用上其实是差不多的，所有这边就只介绍一种，需要使用另外一种时，稍微看看源码就会用了。

创建Cache

PathChildrenCache有6种构造方法，比另外两个加起来还多，这边介绍下常用的参数：

/**
 * @param client           传入client
 * @param path             监视的path
 * @param cacheData        是否缓存data，写false，你会得不到节点的数据。只能拿到stat。
 * @param dataIsCompressed 是否压缩
 */
PathChildrenCache pathChildrenCache = new PathChildrenCache(cf2,"/test",true);
pathChildrenCache.start();

StartMode

PathChildrencache有三种启动模式：(另外两种Cache则没这么丰富)

/**
* The cache will be primed (in the backgrouinitial values.
* Events for existing and new nodes will be posted.
* 
* 默认的启动模式。异步初始化
*/
StartMode.NORMAL
/**
* The cache will be primed (in the foregrouinitial values.
* {@link PathChildrenCache#rebuild()} will be called before
* the {@link PathChildrenCache#start(Stamethod returns
* in order to get an initial view of the node.
* 
* 同步初始化Cache。创建后就从服务器拉数据。
*/
StartMode.BUILD_INITIAL_CACHE
/**
* After cache is primed with initial values (in the background) a
* {@link PathChildrenCacheEvent.Type#INITIALIZED} will be posted.
* 
* 异步初始化，Cache准备好后，会触发INITIALIZED类型的事件。Listerner会收到该通知，
* 通过pathChildrenCacheEvent.getType()就能拿到。
*/
StartMode.POST_INITIALIZED_EVENT

其实就是一个异步/同步初始化、是否有通知的区别。

Listerner

通过给Listerner设置监听器，我们就能够对设置的节点进行监听工作。

//设置为true才能够达到节点的data。
PathChildrenCache pathChildrenCache = new PathChildrenCache(cf2,"/test",true);
pathChildrenCache.start();

pathChildrenCache.getListenable().addListener((curatorFramework, pathChildrenCacheEvent) -> {
    String nodeName = ZKPaths.getNodeFromPath(pathChildrenCacheEvent.getData().getPath());
    String path2 = pathChildrenCacheEvent.getData().getPath();

    switch (pathChildrenCacheEvent.getType()){
        case CHILD_ADDED:
            System.out.println(nodeName+" :add: "+path2);break;
        case CHILD_REMOVED:
            System.out.println("remove");break;
        case CHILD_UPDATED:
            System.out.println("changed");
            System.out.println(new String(pathChildrenCacheEvent.getData().getData()));
            break;
    }
});

我们能够获取到节点对type，并作出对应对处理。
